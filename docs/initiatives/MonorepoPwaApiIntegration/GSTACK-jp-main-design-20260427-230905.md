## Tier 1 — Monorepo Architecture Principles

This tier has two kinds of content:

- **Stable principles** (Subtree Pattern, Shared Package Strategy, DX Philosophy): these
  hold across every future addition. Change them only with deliberate justification.
- **Evolving roadmaps** (Domain Evolution Path, Infrastructure Tiers, Client Surface Area):
  these are directional, not prescriptive. They will change as the project learns.
  Reference them for orientation; update them when the direction shifts.

Reference this section before adding a new stack, client, or infra component.

### Purpose

A monorepo of deployable skeleton projects using a TODO app as the domain vehicle.
Each skeleton demonstrates a distinct tech stack or deployment pattern. The domain
evolves incrementally: TODO → Task Management → Retrospective Platform.

Two audiences:

- **Future-self**: a reliable, battle-tested starting point for new projects
- **Hiring engineers**: demonstrates architectural judgment, breadth, and system thinking

### Client Surface Area

Planned client types (order reflects current priority, not fixed roadmap):

- Web — Vite PWA (exists)
- Web — Next.js SSR (richer SSR-capable web client — next client after Phase 1 stabilizes)
- Desktop — Electron (further out)
- Mobile — iOS + Android (Expo/React Native, single codebase)

New clients are added when a feature or surface area warrants it, not on a fixed schedule.

### Subtree Pattern

Each app lives in its own upstream repo AND in this monorepo via `git subtree`.

**Why**: Each skeleton must be independently clonable and usable without the monorepo.
The monorepo is the composition and orchestration layer — not the only way to use any
individual skeleton.

**Implication**: Apps in `apps/` are not modified directly in the monorepo except to
test integration. Changes go to the upstream repo first, then sync in.

**Sync workflow**: Each app has a GitHub Actions workflow that pulls upstream changes
and opens a PR (mirrors the existing `todo-pwa-vite` sync pattern).

### Shared Package Strategy

Shared code lives in `packages/` in the monorepo root. It does NOT live in any upstream
subtree repo.

Packages are published (npm or GitHub Packages) so upstream repos can consume them as
normal dependencies. During monorepo development, pnpm workspaces links them locally
without requiring a publish step. After publishing, upstream repos switch to the
versioned npm dependency. This transition is a tracked step, not automatic.

Phase 1 does not require automated version bump propagation in upstream repos — manual
install of the published package version is acceptable. Automation is deferred.

**Principle**: The monorepo owns the shared contract. Upstream repos are consumers.

Current planned packages:

- `packages/todo-types` — shared TypeScript types and Zod schemas for the todo domain

### Domain Evolution Path

_(Evolving roadmap — phases beyond 2 are directional, not prescribed. New intermediate
phases will be defined as the surface area and feature set becomes clear.)_

The TODO domain is intentionally trivial so infrastructure and architectural patterns
are the visible signal. Domain complexity grows incrementally:

1. **Domain Phase 1** (now): Basic CRUD todos, offline-first + API-backed
2. **Domain Phase 2**: Lists, tags, due dates, priorities
3. **Domain Phase 3+**: Direction — multi-user, real-time sync, task management, shared
   lists, assignments, retrospectives. Exact phases defined incrementally.

**End state direction**: Full task management and retrospective platform.

New domain phases begin when the current infra and DX patterns are stable enough to
carry the additional complexity without learning two things at once. Phases are scoped
just-in-time, not pre-specified.

### Infrastructure Tiers

_(Evolving roadmap — the production tier will change as the project matures.)_

Three deployment tiers are co-prioritized. Each tier must work independently so the
system can be run at any level of infrastructure investment:

**Tier A — Development (local)**

- Individual apps: `pnpm dev` per app, or `turbo run dev` from monorepo root
- All services together: `docker compose up` (existing `docker-compose.yml`)
- Target: any developer machine

**Tier B — Self-Hosted / Staging (Docker Compose) [PRIMARY DISTRIBUTION TARGET]**

- Same `docker-compose.yml` (or a `docker-compose.prod.yml` variant) deployed to any
  compute capable of running Docker Compose: a VPS, home server, or cloud VM
- No orchestration layer required — just Docker and a public IP or Cloudflare Tunnel
- This is the primary staging and demo target — more important to get right than
  cloud production for Integration Phase 1. A working Docker Compose environment
  proves the two services compose correctly before deploying to Railway.
- PWA nginx proxy handles `/api/` → NestJS internally — no cross-origin concerns in Docker Compose
- SQLite data persists in a named Docker volume between `docker compose down/up`
- Target: any single host with Docker installed

**Tier C — Production Cloud**

- _Now_: Railway (API, SQLite) + Cloudflare Pages (PWA)
- _Later_: Self-hosted Kubernetes cluster exposed via Cloudflare Tunnels
  - SQLite → Postgres at k8s migration (Prisma provider swap, schema unchanged)
  - Cloudflare Tunnel replaces Railway's public URL
- Target: production workloads, scalable, ops-managed

**IaC**: Pulumi manages Cloudflare resources ONLY (DNS, Pages, Tunnels). Railway is
NOT managed by Pulumi — it uses its own GitHub integration or `railway up`. k8s
manifests and Helm charts live in `infra/k8s/` and `infra/helm/`. Docker Compose
files live at the monorepo root.

### DX Philosophy

_(Stable principle — this hierarchy governs all planning work.)_

**Visible communication is non-negotiable.** Planning lives in GitHub, not in heads.
Every capability has a traceable path from platform use case → component feature → implementation spec.

**Three-tier planning hierarchy:**

```
Initiatives                    ← docs/initiatives/{initiative}/
  └── Implementation specs      ← docs/specs/{initiative}/
        └── Feature docs        ← docs/features/[STATUS]GH{n}_{slug}.md
```

1. **Initiatives** — Platform-level use cases drawn from the system PRD
   (`docs/project/Project BootStrap Mono Repo - PRD.md`). Each component in the
   PRD's "General Outline" (Vite PWA, NestJS API, Next.js SSR, iOS, Android, Astro,
   shared types, infra) maps to one or more initiatives. Discovery and design
   context live in `docs/initiatives/{initiative}/`.

2. **Implementation specs** — superpowers plans for a specific initiative.
   Commit these under `docs/specs/{initiative}/`, using the same initiative
   folder name as `docs/initiatives/{initiative}/`.

3. **Features** — Feature docs in `docs/features/` are implementation context.
   Agents read the active feature/spec docs at the start of every development
   session.

The `docs/specs/{initiative}/` folder holds implementation plans/specs for each
initiative. GitHub Issues are written manually from accepted initiatives and
specs when issue tracking is useful. Automated issue sync is deprecated.

**Other DX principles:**

- Patterns are learned incrementally — complexity is added when the previous layer is stable
- Each new skeleton or infra addition documents the "why" in a feature doc
- CI/CD mirrors what a small engineering team would actually use
- Boilerplate quality means: clone it, run one command, have something working

## Tier 2 — Integration Phase 1: Two-Component Integration

### Problem Statement

`todo-api-nestjs` exists as a git subtree but has no sync workflow, no deployment
target, and no runtime connection to the PWA. The monorepo has two components that
don't yet compose. Integration Phase 1 makes the NestJS API a first-class citizen
and establishes the shared types pattern.

### What Makes This Cool

- The monorepo becomes a real system, not two apps in a folder
- A published shared types package demonstrates the "monorepo as platform" pattern
- The dual-mode PWA shows offline-first AND API-backed in one codebase — architectural
  range that's rare in a portfolio piece
- The Helm chart scaffold sets up the k8s end-state from day one, even before the
  cluster exists

### Premises

1. Git subtree is correct — each skeleton stays independently clonable
2. The PWA supports both modes — offline-first (localStorage) and API-backed (env-toggled)
3. Railway now (SQLite), k8s later (Postgres) — Prisma provider swap is the migration path
4. `packages/todo-types` is published from the monorepo, consumed by upstream repos
5. k8s manifests + Helm chart are developed concurrently with Railway deployment
6. Cloudflare Tunnels handle k8s ingress — ties back to existing Cloudflare/Pulumi investment

### Approaches Considered

#### Approach A: Infra-Only Integration

Give the NestJS API monorepo parity (sync workflow, Pulumi, CI/CD) but keep both apps
independent at runtime. PWA stays offline-first only.

- Effort: M | Risk: Low
- Pros: Clean separation, no subtree coupling, simpler PWA
- Cons: Two apps that don't talk feels like two repos, not a system

#### Approach B: Full Runtime Integration

Replace the PWA's localStorage/worker approach with API calls entirely.

- Effort: L | Risk: Med
- Pros: Complete full-stack story, shared types essential
- Cons: Breaks offline-first capability, couples subtrees at code level

#### Approach C: Offline-First PWA with API Sync ✓ CHOSEN (updated by eng review)

NestJS gets full infra parity. PWA is offline-first by default — always works locally,
syncs to the API when the network is available. Single build, no env-var mode toggle.

- Effort: L | Risk: Med
- Pros: Best showcase — demonstrates real offline-first pattern (not just a toggle);
  shared types is the enterprise signal; each upstream repo stays independently usable
- Cons: Requires a write queue and flush-on-reconnect handler; API must accept
  client-generated UUIDs for idempotent creates

### Recommended Approach

**Approach C: Offline-First PWA with API Sync**

One build. The PWA always works locally using the Web Worker + localStorage. When the
network is available it syncs to the NestJS API. Offline writes are queued and flushed
automatically on reconnect (last-write-wins — correct for a single-user todo app).

This is the real offline-first pattern. It demonstrates that you understand both the
offline and API layers and integrated them into a coherent UX — not two separate modes
that a user switches between. The shared types package is the concrete artifact that
makes this a genuine monorepo platform rather than two apps in a folder.

**D5 eng-review decisions (2026-04-28):**

- `VITE_TODO_MODE` env var eliminated — single build variant only
- Single `useTodos` hook abstracts data source; `useTodoWorker` and `useTodoApi` are
  internal implementations both satisfying the `TodoHook` interface
- `useTodos` assigned at module level (not conditional inside hook — satisfies React rules)
- `offline: boolean` in hook return = runtime `navigator.onLine` state (not build-time flag)
- S2 sync strategy: write queue in localStorage, flush-on-reconnect, last-write-wins
- Client-generated UUIDs (crypto.randomUUID()) required for idempotent queue flush
- `CreateTodoDto` must accept optional `id?: string` so client IDs are preserved on API write
- `VITE_TODO_API_URL` retained — still needed to point the API client at the right host
